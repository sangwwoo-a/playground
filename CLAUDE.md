## 🚨 최우선 규칙 — HTML/대용량 파일 편집

### 절대 하지 말 것

1. **❌ Edit 도구로 큰 string 한 번에 교체 금지** (300줄 이상 또는 5000자 이상)
   - 문자열이 잘려서 파일 끝부분이 사라지는 사고 반복 발생
   - HTML script 태그가 끊겨서 화면에 코드가 텍스트로 노출되는 문제 발생

2. **❌ Edit로 setup-modal, body 같은 대형 블록 통째로 교체 금지**
   - 잘림 → `</script>` 사라짐 → HTML 파싱 깨짐

3. **❌ 잘린 파일에 cat append로 이어 붙이기 금지**
   - 중간이 누락된 채 새 코드가 추가되어 코드 누수 발생
   - 같은 함수가 두 번 들어가거나, 따옴표 깨진 상태로 결합됨

### 반드시 할 것

1. **✅ 큰 변경은 5~7개 청크로 나눠 `cat >> heredoc` 으로 작성**
   - 청크 1개당 70~100줄 이하 (한글 포함이라 더 짧게)
   - 각 청크 끝에 `echo "C1 OK"` 같은 표식 출력해 진행 확인

2. **✅ 새로 만들 때는 빈 파일로 시작 (`> file`) 후 청크 단위 append**
   - 잘린 파일에 덧붙이지 말고, 처음부터 새로 쓰기

3. **✅ 청크 추가 후 매번 검증 — 5가지 체크 (함수 중복 검사 필수!)**
   ```bash
   # JS syntax (인라인 script 부분)
   node -e "const html=...; const code=html.substring(...); new vm.Script(code)"

   # 닫는 태그 정상
   grep -c "</script>" file
   grep -c "</body>" file
   grep -c "</html>" file

   # body 본문에 코드 누수 없음 (script 태그 제외하고 검사)
   const cleanBody = body.replace(/<script>[\s\S]*?<\/script>/g, '');
   cleanBody.includes('avatarsHtml') === false

   # 파일이 끝까지 정상 (마지막 줄이 </html>)
   tail -3 file

   # 함수 중복 정의 검사 — 이게 빠지면 화면에 코드 노출 발생
   // 모든 핵심 함수가 정확히 1번만 정의되었는지 확인
   for fn in functionNames {
     const c = (code.match(new RegExp('function ' + fn + '\\\\b', 'g')) || []).length;
     if (c !== 1) FAIL;
   }
   ```

4. **✅ 작은 수정만 Edit 사용** — 5~10줄 이내 변경
   - 한 줄 함수 시그니처 변경
   - 변수명 교체
   - 단일 if 블록 수정

  - 핀 좌표는 그 주소 놀이터들의 좌표 중앙값 또는 첫 번째 좌표 사용
  - 카카오 지오코딩으로 도로명 주소 → 좌표 재계산도 옵션 (정확도 ↑)

## 🧮 놀이터 그룹화 알고리즘 디테일 (구현 시 참고)

### 1. 그룹 키 결정 — `groupKey()` 함수

**도로명 주소를 정규화한 문자열**을 그룹 키로 사용. 정규화 규칙:

```javascript
function groupKey(addr) {
  if (!addr) return null;
  return addr
    // 1) 끝의 (산곡동) 같은 행정동 괄호 제거
    .replace(/\s*[\(（][^\)）]*[\)）]\s*$/, '')
    // 2) 한글 공백/특수공백 통일
    .replace(/\s+/g, ' ')
    // 3) "○로 ○○-○" 같은 부번지(-) 정규화: "안남로123번길 46-2" → "안남로123번길 46"
    //    단지 안 동별로 -1, -2 붙는 경우 합치려는 의도
    .replace(/(\d+)-\d+(?=\s|$)/, '$1')
    .trim();
}
```

**주의**:
- 부번지 제거(`46-2` → `46`)는 단지 그룹화에 도움이 되지만, 빌라/연립처럼 진짜 다른 건물이 합쳐질 수도 있음
- → 출시 후 사용자 피드백 보고 조정
- 학교/공원은 보통 부번지 없으니 영향 적음

### 3. 예약(약속) 데이터 — 그룹이 아닌 개별 놀이터에 귀속

- **그룹은 표시(UI) 단위**, **예약은 개별 놀이터 단위**
- "오늘 N명"이 그룹 핀 위에 표시될 때 = 그룹 안 모든 놀이터의 예약 합계
- 핀 클릭 → 리스트에서 선택한 개별 놀이터에 약속 등록
- 단일 사용자가 같은 그룹의 다른 놀이터에 중복 등록 가능 (각각 별개로 카운트)
  - 예: "401동 앞" 14-16시 + "405동 앞" 14-16시 → 사용자 본인이 의도한 것이므로 허용
  - 단, 같은 놀이터 내 중복은 여전히 차단 (`r.mine` 필터)

### 4. 좌표 계산 — 중앙값(median) 사용

```javascript
function groupCenter(items) {
  // 평균 대신 중앙값 — 이상치(엉뚱한 좌표) 영향 적음
  const lats = items.map(p => p.y).sort((a, b) => a - b);
  const lngs = items.map(p => p.x).sort((a, b) => a - b);
  const mid = Math.floor(items.length / 2);
  return {
    lat: items.length % 2 ? lats[mid] : (lats[mid-1] + lats[mid]) / 2,
    lng: items.length % 2 ? lngs[mid] : (lngs[mid-1] + lngs[mid]) / 2,
  };
}
```

**옵션**: 카카오 Geocoder로 도로명 주소 → 좌표 한 번 더 계산해서 비교, 차이 작으면 카카오 결과 사용 (더 정확).

### 5. 그룹명 자동 추출 — 사용자에게 표시할 이름

도로명 주소만 보여주면 어느 단지인지 모를 수 있음. items의 `n`(시설명)에서 공통 부분 뽑기:

```javascript
function groupTitle(items) {
  if (items.length === 1) return items[0].n;
  // 첫 항목의 시설명에서 "401동 앞 놀이터" 같은 후미 제거
  // 예: "산곡 우성4차아파트 401동 앞 놀이터" → "산곡 우성4차아파트"
  const names = items.map(p => p.n);
  // 공통 prefix 길이 찾기
  let common = '';
  for (let i = 0; i < names[0].length; i++) {
    const ch = names[0][i];
    if (names.every(n => n[i] === ch)) common += ch;
    else break;
  }
  // 공통 prefix가 너무 짧으면 전체 도로명 주소 사용
  return common.trim().length > 5 ? common.trim() : items[0].a.replace(/\s*\(.*?\)\s*$/, '');
}
```

### 6. UI 흐름 — 핀 → 리스트 → 시간대

**그룹 핀 (items.length >= 2)**:
- 핀 위 라벨: `${groupTitle} [${items.length}]` 또는 `오늘 ${total}명`
- 클릭 → **그룹 리스트 모달** (새 모달 추가 필요)

**그룹 리스트 모달 구성**:

│                              │

### 7. 코드 변경 위치 — 어디를 수정해야 하나

**`filterAndDisplay()`** 함수 수정:
- 거리 계산 + 정렬 후, **그룹화 단계 추가**
- `state.playgrounds = groups.slice(0, MAX_DISPLAY)` 형태
- 각 group은 `{id, key, name, addr, lat, lng, items, reservations(파생)}` 구조

**`renderMarkers()`** 수정:
- 그룹 핀 라벨에 카운트 표시 (items.length >= 2일 때)
- 시각적으로 단일 핀과 약간 구분 (예: 카운트 배지 추가)

**`openModal(id)`** 분기:
- group.items.length === 1 → 시간대 모달 (기존 흐름)
- group.items.length >= 2 → **새 함수** `openGroupListModal(id)` 호출
  - 리스트에서 항목 클릭 시 `openModal(realPgId)` 호출 (시간대 모달)

**`countTotal(group)`** 수정:
- 그룹의 모든 items의 reservations 합계 (item별 reservations 유지)

### 8. 마이그레이션 — 기존 코드와의 호환

기존 `state.playgrounds`는 평탄한 배열이었음. 그룹화 후엔 각 entry가 `items` 배열 보유.
**기존 단일 놀이터 케이스**도 `items: [self]` 형태로 통일하면 분기 줄어듦. 즉:
- 단일이든 복수든 `playgrounds[i].items` 항상 존재
- `items.length === 1`일 때만 UI 흐름 단순화

### 9. 데이터 갱신 시점

- 그룹화는 **앱 로드 시 1회** 실행 (8만 곳 → 그룹 N개로 줄임)
- 그룹 결과 캐시는 `state` 메모리에만 (localStorage 안 씀)
- 새로고침 시마다 다시 계산 (단순함, 성능 충분)

- 예: "새벽 1시네요, 푹 쉬세요" → 사용자가 이미 자고 일어난 상태일 수 있음
- 사용자가 직접 말한 정보만 활용, 그 외엔 작업에만 집중
